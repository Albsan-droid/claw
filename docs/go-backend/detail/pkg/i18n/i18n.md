# pkg/i18n/i18n.go 詳細設計

## 対象ソース
- `pkg/i18n/i18n.go`

## 概要
`i18n.go` は Go バックエンドの最小ローカライズ基盤を提供する。メッセージカタログの登録、ロケール正規化、キー検索、書式付き翻訳取得を担い、各 `messages_*.go` の `init()` から呼ばれる。

## 責務
- ロケール別メッセージ辞書の保持
- 辞書マージ登録
- `Accept-Language` / ロケールタグの正規化
- 英語フォールバックを含む翻訳取得

## 主要な型・関数・メソッド
- `var messages = map[string]map[string]string{}`
- `func register(locale string, m map[string]string)`
- `func NormalizeLocale(locale string) string`
- `func T(locale, key string) string`
- `func Tf(locale, key string, args ...interface{}) string`

## 詳細動作
### 1. メッセージ保持
- `messages` は `map[locale]map[key]translation` 形式のグローバル辞書。
- `register()` は対象ロケールの map が未作成なら初期化し、受け取った `m` の各 key/value を上書きコピーする。
- 複数ファイルから同じ locale に対して段階的に登録する設計。

### 2. `NormalizeLocale()`
- 入力文字列を `strings.TrimSpace()` する。
- 空文字なら `"en"` を返す。
- `Accept-Language` 形式 (`ja, en;q=0.9`) は最初の要素だけ取り出す。
- 品質値 `;q=...` は `;` 以降を削る。
- `ja-JP` / `ja_JP` のような派生タグは `-` または `_` の前半だけ使う。
- 最後に小文字化して返す。

### 3. `T(locale, key)`
- まず `NormalizeLocale(locale)` を適用する。
- `messages[locale][key]` があればそれを返す。
- 見つからず locale が `en` 以外なら `messages["en"][key]` を試す。
- それでも無ければ key 文字列自体を返す。

### 4. `Tf(locale, key, args...)`
- `T(locale, key)` の戻り値を `fmt.Sprintf()` へ渡すだけの薄いラッパ。
- `%s` など埋め込み付きメッセージ専用。

## 入出力・副作用・永続化
- 入力
  - locale 文字列
  - 翻訳キー
  - 任意の format 引数
- 出力
  - ローカライズ済み文字列
- 永続化
  - なし
- 副作用
  - `register()` によるグローバル辞書更新

## 依存関係
- 標準ライブラリ: `fmt`, `strings`
- カタログ提供元: `pkg/i18n/messages_agent.go`, `messages_channel.go`, `messages_config.go`, `messages_status.go` など

## エラーハンドリング・制約
- 辞書アクセスはエラーを返さない。
- 未登録キーは key 文字列そのものが表示されるため、UI 上に内部キーが露出し得る。
- ロケールは 2 文字言語コード前提で、`zh-Hant` のような script/region 詳細は保持しない。
- グローバル `messages` への登録は同期化していないが、通常は package init 時の一方向登録のみを前提にしている。
